9. [Pod Template vs Deployment manifest, Ingress vs Loadbalancer vs ClusterIP vs NodePort] 

In Kubernetes, both Pod and Deployment manifest files are used to describe how applications should be run within the cluster, but they serve different purposes and contain different configurations. Here’s a detailed comparison of the two:
 
### Pod Manifest File
 
A Pod is the smallest deployable unit in Kubernetes. It represents a single instance of a running process in the cluster and can contain one or more containers.
 
**Pod Manifest Example**:
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
spec:
  containers:
  - name: my-container
    image: nginx:latest
    ports:
    - containerPort: 80
```
 
**Key Characteristics**:
- **Single Unit**: Defines a single Pod, which might contain multiple containers running on the same node and sharing the same network namespace.
- **Static**: Pods defined this way are static and don’t provide features like scaling, rolling updates, or self-healing.
- **Use Cases**: Typically used for simple or testing scenarios, where advanced features are not required.
 
### Deployment Manifest File
 
A Deployment provides declarative updates for Pods and ReplicaSets. It’s a higher-level concept that manages the lifecycle of applications, providing capabilities like scaling, rolling updates, and rollback.
 
**Deployment Manifest Example**:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:latest
        ports:
        - containerPort: 80
```
 
**Key Characteristics**:
- **Declarative Updates**: Allows you to declare the desired state, and Kubernetes manages the changes to reach and maintain that state.
- **Scaling**: Supports horizontal scaling (adjusting the number of replicas).
- **Rolling Updates**: Enables rolling updates, ensuring zero downtime by gradually updating instances of the application.
- **Rollback**: Facilitates rollback to previous versions if an update fails.
- **Self-Healing**: Automatically replaces failed or unhealthy Pods to maintain the desired number of replicas.


Ingress

->Ingress is a native Kubernetes resource like pods, deployments, etc. Using ingress, you can maintain the DNS routing configurations. The ingress controller does the actual routing by reading the routing rules from ingress objects stored in etcd.
->Ingress is not a load balancer. It contains all the routing rules, custom headers, and TLS configurations. The ingress controller acts as a load balancer.
->Ingress is a kubernetes object. Nginx is used as an ingress controller (Reverse proxy).

Egress

->In Kubernetes, Egress refers to the traffic that flows out of a cluster, from a pod to an external endpoint. Egress traffic can be used to access external services such as databases, APIs, and other services running outside of the cluster.

Cluster IP
->In Kubernetes, the ClusterIP Service is used for Pod-to-Pod communication within the same cluster. This means that a client running outside of the cluster, such as a user accessing an application over the internet, cannot directly access a ClusterIP Service.

Node Port
->The NodePort Service is a way to expose your application to external clients. An external client is anyone who is trying to access your application from outside of the Kubernetes cluster.

Loadbalancer
->A LoadBalancer Service is another way you can expose your applications to external clients. However, it only works if you're using Kubernetes on a cloud platform that supports this Service type.

25. what are advantage and disadvantage of multi stage build in docker.

-> **Multi-stage builds** in Docker provide a way to optimize Docker images by allowing the creation of separate stages within a single `Dockerfile`. Each stage can contain distinct steps for building, testing, or preparing an application, and only the necessary output is carried over to the final image. This method is commonly used to produce smaller, more efficient, and secure production images, as unnecessary dependencies and build tools are excluded from the final image. 

Here are some **advantages and disadvantages** of using multi-stage builds:

---

### Advantages of Multi-Stage Builds

1. **Smaller Final Image Size**:
   - Only essential files and dependencies are copied to the final image, reducing its size. For instance, in a typical web application, only the application binary and static assets may be included in the final image, leaving out compilers and other build tools.
   - Smaller images download and deploy faster, which can improve CI/CD and deployment times.

2. **Enhanced Security**:
   - Since unnecessary tools and dependencies (like compilers or build tools) are left out of the final image, the attack surface is reduced.
   - Fewer components in the production image mean fewer potential vulnerabilities.

3. **Improved Build Efficiency**:
   - Different stages can reuse cache layers, which speeds up the build process by not recompiling or rebuilding parts that haven’t changed.
   - Using multi-stage builds, you can divide the Dockerfile logically and re-build only the stages that require updates, saving time and resources.

4. **Simplified Dockerfile and Maintainability**:
   - Multi-stage builds allow for a more modular and organized approach within the Dockerfile, making it easier to understand and maintain.
   - Different stages can be tailored specifically for build, test, and deployment needs, reducing the complexity of handling dependencies for each stage separately.

5. **Improved CI/CD Workflow**:
   - Multi-stage builds enable the integration of testing and linting steps into the Dockerfile, allowing these processes to be handled within the container, which improves consistency and reproducibility across different environments.
   - CI/CD pipelines can use a single Dockerfile for all steps, from development through production, improving efficiency.

---

### Disadvantages of Multi-Stage Builds

1. **Increased Complexity for Simple Projects**:
   - For small applications or simple projects, multi-stage builds can add unnecessary complexity to the Dockerfile, making it harder to follow.
   - Additional stages may not add much value for basic applications that don’t have many dependencies or build steps.

2. **Limited Layer Caching**:
   - Docker layer caching only applies up to the point where a change occurs. If a change happens early in the build process (like in the first stage), subsequent stages will need to be rebuilt, increasing build times.
   - This can be an issue when large dependencies change frequently, as the entire multi-stage build process may need to re-run, even if the final stages remain the same.

3. **Increased Build Time in Certain Scenarios**:
   - For complex applications with multiple stages and dependencies, the build process can take longer because each stage might add build time overhead.
   - If the Dockerfile structure isn’t optimized, it can lead to slower builds compared to single-stage Dockerfiles, especially if many dependencies change frequently.

4. **Compatibility Issues**:
   - Multi-stage builds are only supported in Docker 17.05 and newer, which may be a limitation if your deployment environment uses older Docker versions.
   - Some CI/CD platforms or orchestration tools may have limitations or lack compatibility with multi-stage builds.

5. **Learning Curve**:
   - Developers need to understand how multi-stage builds work to use them effectively. This can be a challenge for teams unfamiliar with Docker best practices, especially if they need to balance stages, caching, and layer management for optimal builds.
   - Multi-stage builds also require careful structuring to optimize caching, which may not be straightforward without experience.

---

### When to Use Multi-Stage Builds
Multi-stage builds are especially valuable for:
- **Microservices and modular applications**: where lightweight and efficient images are beneficial.
- **Large applications with many dependencies**: where reducing the image size and isolating build tools is critical for security and performance.
- **CI/CD environments**: that require reproducible builds across various stages (build, test, deploy).

For small applications or where Docker builds are very simple, single-stage builds might suffice without added complexity. 

Overall, multi-stage builds provide significant benefits for producing optimized, secure, and modular Docker images, particularly for production environments.

27. Difference between bridge and host networking in docker?

-> Docker networking allows containers to communicate with each other, and with the host system, using various networking modes:
   - **Bridge (default)**: Each container gets its own IP address on a private Docker network, and containers on the same network can communicate with each other.
   - **Host**: The container shares the host’s network stack, meaning it doesn’t get its own IP address, but instead uses the host’s IP and ports.
   - **None**: The container has no network interface. It's isolated from other containers and the host.
   - **Overlay**: Used to connect containers across multiple Docker hosts in a Swarm cluster, forming a virtual network.
   - **Macvlan**: Assigns a MAC address to each container, making them appear as physical devices on the network.

40. What is controller in k8s?

->In Kubernetes, a **Controller** is a control loop that continuously monitors the state of the cluster and ensures that the desired state matches the actual state. If the actual state deviates (e.g., a pod crashes or a node goes offline), the controller takes corrective actions to reconcile the difference, such as restarting pods or creating new resources.

### **Types of Controllers in Kubernetes**
Here are some of the most common controllers used in Kubernetes:
### 1. **ReplicationController**
- Ensures a specified number of **pod replicas** are running at any given time.
- If a pod fails, the controller will create a new one to replace it.
### 2. **ReplicaSet**
- A newer version of **ReplicationController** with more flexible selectors.
- Ensures a specified number of replicas are running.
### 3. **Deployment**
- The most common controller used for **managing stateless applications**.
- Provides rolling updates,. rollbacks, and declarative updates to pods and ReplicaSets.
### 4. **StatefulSet**
- Manages **stateful applications** such as databases.
- Provides guarantees around **pod identity and ordering** (e.g., pod names are persistent and ordered).
### 5. **DaemonSet**
- Ensures that **one instance of a pod runs on each node** in the cluster.
- Useful for **logging agents, monitoring agents, or networking tools** (e.g., Fluentd, Prometheus Node Exporter).
### 6. **Job**
- Ensures that a **batch job completes successfully** (i.e., runs a task to completion).
### 7. **CronJob**
- **Schedules jobs to run periodically** based on a cron schedule (e.g., daily backups).


41. Probes in k8s

-> In k8s there are three type of probes.

## **1. Liveness Probe**
- **Purpose**: Checks if the container is alive or running.
- **Action**: If the probe fails, Kubernetes restarts the container because it assumes the container has crashed or is in a non-recoverable state.
**Explanation:**
- **`httpGet`**: The probe sends an HTTP GET request to `/` on port 80.
- **`initialDelaySeconds`**: The probe waits 5 seconds after the container starts before the first check.
- **`periodSeconds`**: The probe checks the container every 10 seconds.

## **2. Readiness Probe**
- **Purpose**: Checks if the container is ready to **serve traffic**.
- **Action**: If the probe fails, Kubernetes will stop sending traffic to this container, but it won’t restart it. This is useful when a container needs time to initialize or temporarily becomes unavailable.
**Explanation:**
- **`readinessProbe`**: Similar to the liveness probe, but it ensures the container is ready to serve requests.
- When the probe fails, the container is **removed from the service endpoints** until it passes the probe again.

## **3. Startup Probe**
- **Purpose**: Used to check if the **container has finished starting**. This is useful when a container takes a long time to initialize. Once the startup probe succeeds, Kubernetes stops performing other probes (like liveness or readiness probes).
- **Action**: If the probe fails, the container is restarted.
**Explanation:**
- **`startupProbe`**: Ensures the container has fully started before other probes begin.
- **`failureThreshold`**: If the probe fails 3 times, Kubernetes restarts the container.


42. Sidecar & Init container.

-> ## **1. Sidecar Container**  
A **sidecar container** runs alongside the main application container within the same Pod to provide additional functionality. Sidecar containers are commonly used for tasks like logging, monitoring, proxying, or configuration updates.  

### **Use Case of Sidecar Containers**  
- Add logging or monitoring agents (e.g., Fluentd).
- Run proxies (e.g., Envoy, Istio sidecars for service mesh).
- Perform data synchronization tasks between containers.
- Share configuration files dynamically.

## **2. Init Container**  
An **init container** is a type of container that runs **before the main application container starts**. It ensures that preconditions are met before the main container can run (e.g., setting up configuration files, waiting for a service dependency, or loading secrets).

### **Characteristics of Init Containers:**
- They always **run to completion** (they must succeed for the main container to start).
- **Run sequentially**: If there are multiple init containers, each runs one by one.
- They can perform **one-time setup tasks** that don’t need to run continuously.



43. Differnce between copy & add comand in dockerfile.

-> The COPY instruction is straightforward and does exactly what its name implies: It copies files and directories from a source within the build context to a destination layer in the Docker image. This instruction can be used to copy both files and directories, and all paths on the host are relative to the root of the build context.

-> The ADD instruction provides the same functionality that the COPY instruction does, but it also has additional functionality that, if misunderstood, can introduce complexity and potential security risks.

Extended functionality: In addition to copying local files and directories from the build context, ADD provides the following advanced functionality:
Handle URLs: When supplied as a source, the file referenced by a URL will be downloaded to the current Docker image layer at the supplied destination path.
Extract archives: When supplied as a source, ADD will automatically unpack and expand archives to the current Docker image layer at the supplied destination path.

44. If an application endpoint is not accessible that is deployed on aks then what we need to check.

-> If an endpoint on **Azure Kubernetes Service (AKS)** is not accessible, there could be several underlying issues. Here are the key areas you should check systematically to troubleshoot the issue:

### 1. **Pod and Service Health**  
   - **Check Pods**: Ensure the pods are running correctly.
     ```bash
     kubectl get pods -A -o wide
     kubectl describe pod <pod-name>
     ```
     Look for any errors, restarts, or crash loops.

   - **Check Service**: Verify the corresponding Kubernetes Service is correctly exposing the pods.
     ```bash
     kubectl get svc -A
     kubectl describe svc <service-name>
     ```
     Ensure the service type (ClusterIP, NodePort, LoadBalancer) is configured correctly.

### 2. **Ingress/Load Balancer Configuration**  
   - **Ingress Controller Status**: If you are using an Ingress, make sure it is correctly configured.
     ```bash
     kubectl get ingress -A
     kubectl describe ingress <ingress-name>
     ```
     Ensure the Ingress points to the correct backend service and routes.

   - **Load Balancer IP**: For services of type `LoadBalancer`, verify the external IP has been assigned.
     ```bash
     kubectl get svc -A
     ```
     If pending, check Azure Load Balancer configurations and quotas.

### 3. **DNS and Networking Issues**  
   - **DNS Resolution**: If accessing the service by name, ensure the DNS resolution is working properly.
     ```bash
     kubectl exec -it <pod> -- nslookup <service-name>
     ```
     Also, ensure Azure DNS configurations are correct for any external domains.

   - **Network Policies**: Check if any **NetworkPolicy** is restricting traffic.
     ```bash
     kubectl get networkpolicy -A
     ```

   - **Firewall and NSGs**: Ensure no Network Security Group (NSG) or firewall rule in Azure is blocking traffic to the cluster or Load Balancer.

### 4. **Application-Level Logs**  
   - **Pod Logs**: Check the logs of the relevant pods to see if the application is behaving correctly.
     ```bash
     kubectl logs <pod-name> --tail=50 -f
     ```

47. If image creation from dockerfile is failing, how you troubleshoot and find which layer is failing.

-> Troubleshooting a failing Docker image build can be tricky, especially if the error is in a specific layer. Here’s how you can systematically identify and resolve the issue: 

### 1. **Read the Error Message**
   - The build process will typically fail with an error message pointing to the specific line in the Dockerfile.
   - Look at the logs to identify which step or layer caused the failure.

### 2. **Add Debugging Steps**
   - To find out which layer is failing, break the Dockerfile into smaller parts and build it step by step:
     ```bash
     docker build --no-cache --progress=plain .
     ```
     - `--no-cache`: Ensures every layer is rebuilt.
     - `--progress=plain`: Provides detailed logs of each step.

### 3. **Build Incrementally**
   - Manually run the commands in the Dockerfile to isolate the issue:
     ```bash
     docker run -it --rm base_image
     ```
     Replace `base_image` with the image used in the `FROM` instruction. Then, try running the commands one by one from your Dockerfile.

### 4. **Use `docker history`**
   - If the image partially builds, use `docker history` to inspect each layer:
     ```bash
     docker history image_name
     ```
     This command will show the command used for each layer, which can help pinpoint the problematic layer.

### 5. **Check for Common Issues**
   - **Syntax Errors**: A small typo in a `RUN`, `COPY`, or other instructions can cause failures.
   - **Missing Files**: Ensure files referenced in `COPY` or `ADD` exist in the build context.
   - **Network Issues**: If the build involves downloading dependencies, check your network and proxy settings.
   - **Permissions**: Ensure proper file permissions if files are being copied or accessed.

### 6. **Enable Verbose Logging**
   - You can increase the verbosity to get more details about the build process:
     ```bash
     docker build --progress=plain .
     ```

### 7. **Test Layers in Isolation**
   - Use intermediate containers to test each layer:
     ```bash
     docker build --target <intermediate-stage-name> -t debug-image .
     ```
     This builds up to a specific stage in a multi-stage Dockerfile.

### 8. **Review Dockerfile Best Practices**
   - Use `&&` in `RUN` instructions to combine commands. If one command fails, it’s easier to identify.
   - Example:
     ```dockerfile
     RUN apt-get update && apt-get install -y some-package && \
         rm -rf /var/lib/apt/lists/*
     ```
